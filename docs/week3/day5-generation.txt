二、"没有 RAG 的 LLM"——先看看问题长什么样
===========================================

▶ 生活类比：新来的实习生

  你公司新来了一个实习生（LLM），ta 毕业于名校，读过海量书籍，知识面很广。
  但 ta 今天第一天入职，完全不知道你公司的内部规定。

  你问 ta："公司 Wi-Fi 密码是什么？"
  实习生不知道真实答案，但 ta 不好意思说"我不知道"，
  于是 ta 猜了一个："公司 Wi-Fi 密码通常是 admin123，或者贴在路由器背面。"

  听起来合理，但是错的。这就叫"幻觉（Hallucination）"。

▶ 幻觉的根源

  LLM 的知识来自训练数据（互联网上的公开文本），训练结束后就不再更新。
  你公司内部的规定（Wi-Fi 密码、报销流程、年假天数）不在训练数据里，
  LLM 根本不知道这些信息。

  但 LLM 被训练成"总是给出有帮助的回答"，它不会说"我不知道"——
  它会编。听起来很像真的，但你没法验证。

▶ 这就是为什么需要 RAG

  RAG = 在问题发给 LLM 之前，先去资料库里翻一下，把相关资料找出来，
  一起塞给 LLM，然后说："只能基于这些资料回答，别自己编。"

  RAG 给 LLM 装了一个"资料库"，让它从"凭记忆回答"变成"查资料回答"。


三、RAG 但不引用——"答案对，但没法验证"
=========================================

▶ 场景

  你已经做了检索——找到了"年假政策"和"病假规定"两个文档块。
  你把这些块拼接到 Prompt 里，LLM 基于它们回答了问题。

  答案是对的："入职满 3 年享有 10 天年假，请病假需提供医院证明。"

  但用户看到这个答案，心里会想："这个数字靠得住吗？是 LLM 编的还是真有这条规定？"

  用户无法验证。这就是"引用（Citation）"要解决的问题。

▶ 生活类比：论文注释

  你写了一篇论文，说"高血压患病率达到 27.5%"。
  老师看到这个数字，问："这个数据从哪来的？"
  如果你没有标注引用出处，老师只能选择相信你或者不信你。
  如果你标注了 [WHO 2023 全球健康报告, p.45]，老师就可以去查证。

  AI 生成的答案也一样——引用标注让用户能"查证"答案的来源。


四、正确的 RAG 生成 + 引用——标准四步流程
==========================================

▶ 第一步：检索（Retrieve）

  用户问题 → 检索器 → Top-K 文档块
  这一步 Day 2-4 你已经完全掌握了。

▶ 第二步：增强 Prompt（Augment）

  把检索到的文档块填进 Jinja2 模板。
  模板长这样（核心部分）：

  ```
  ## 参考资料
  以下是你回答问题时必须依据的唯一信息来源。
  每条资料前标注了编号，引用时请使用 [1] [2] 等编号。

  [1] 年假政策：入职满 1 年享有 5 天带薪年假，满 3 年 10 天...
  [2] 病假规定：请病假需提供二级以上医院开具的病假证明...

  ## 回答要求
  1. 只能基于以上参考资料回答问题，禁止使用你自己的外部知识
  2. 每个关键事实陈述后面必须标注来源编号，例如 [1]、[2][3]
  3. 如果参考资料中没有相关信息，直接说"根据现有资料无法回答"
  ```

  这个模板做了三件事：
    ① 告诉 LLM 资料在哪（[1] [2] 标注的文档块）
    ② 约束 LLM 必须引用（"每个关键事实后面必须标注来源编号"）
    ③ 兜底规则（"没有相关信息就说无法回答"）

▶ Prompt 设计的关键——为什么这些指令要放在 Prompt 里而不是代码里？

  你可能会想：为什么不写 Python 代码来"强制"LLM 引用？
  比如用正则检查输出有没有 [1]，没有就重新生成？

  可以，而且生产环境确实会这样做。但这只能保证"有引用"，
  不能保证"引用是对的"。真正的约束来自 Prompt 设计：

  - "只能基于参考资料回答" → 限制信息来源
  - "每个关键事实必须标注来源" → 要求引用行为
  - "无法回答时说不知道" → 兜底策略

  这叫"软约束"——不是代码层面的强制，但设计良好的 Prompt 能让 LLM 大概率遵守。

▶ 第三步：生成（Generate）

  LLM 收到拼接好的 Prompt，基于资料生成答案。

  示例输出：
  ```
  入职满 3 年的员工享有 10 天带薪年假 [1]。
  年假需提前一周向直属上级申请 [1]。
  关于病假，需提供二级以上医院开具的病假证明 [2]。
  紧急情况可在 24 小时内补交证明 [2]。
  ```

  注意 LLM 的行为：
  - 同一个编号 [1] 引用了多次（两次提到年假，都标 [1]）
  - [2] 也是多次引用
  - LLM 自己做了"信息整合"——把两条资料的内容按逻辑组织起来

▶ 第四步：引用解析（Cite）

  用正则表达式从答案中提取 [1] [2] 等编号，建立"编号 → 原文档"的映射。

  ```python
  # 输入：LLM 回复（含 [1] [2]）
  # 输出：引用列表 [{"number": 1, "text": "年假政策原文..."}, ...]
  import re
  cited = set(int(m.group(1)) for m in re.finditer(r"\[(\d+)\]", answer))
  ```

  这一步的意义：
    用户看到 [1]，点击就能看到原文。
    答案和引用分离——答案保持可读，引用提供可验证性。


五、引用格式的演化——[1] 不是唯一选择
======================================

▶ 为什么是 [1] 而不是 (1) 或 ¹？

  [1] 是最广泛使用的学术引用格式，几乎所有论文都这么用。
  它简洁、无歧义、正则好匹配。

  生产环境中你会看到：

  格式           例子                            适用场景
  ─────────────────────────────────────────────────────────
  [N]           这是规定 [1]                      最简单的，你的 Demo 用这个
  [来源: N]     这是规定 [来源: 1]                 可读性更好
  [N](URL)      这是规定 [1](doc-5链接)            可点击跳转
  脚注格式       这是规定¹（页面底部列出来源）       长篇报告

  你现在用 [N] 就够了。做 Week 4 项目时可以升级为可点击的 [N](链接) 格式。


六、边界情况——RAG 也不是万能的
================================

▶ 边界 1：资料里没有答案

  用户问："公司附近有什么好吃的餐厅？"
  知识库里是公司制度文档，没有餐厅信息。

  检索结果相似度很低（比如都在 0.3 以下），因为确实没有相关文档。

  处理方式：
    ① 相似度阈值过滤：sim < 0.6 的文档块丢弃，不进入 Prompt
    ② Prompt 兜底指令："参考资料中没有相关信息时，说无法回答"

  LLM 的预期输出：
  "根据现有资料无法回答您的问题。当前知识库涵盖公司报销、请假、
  IT、入职、薪酬等制度信息，不包含周边餐厅推荐。"

  这就是 Prompt 模板中那条"无法回答"规则起作用的地方。

▶ 边界 2：多条资料信息冲突

  如果两条资料对同一个问题的回答不一致怎么办？

  比如资料 A 说"年假 5 天"，资料 B 说"年假 10 天"。
  （这通常是知识库没有及时更新导致的，不同版本混在一起了。）

  处理方式：
    ① LLM 应该同时引用两条，并说明差异
       "根据资料 A [1]，年假为 5 天；根据资料 B [2]，年假为 10 天。
        两处信息不一致，建议以最新版员工手册为准。"
    ② 更好的做法：在检索阶段就处理——给文档加"更新时间"元数据，
       检索时优先返回最新的版本。

  这种"多文档推理"是高级 RAG 的内容，Week 7-8 会深入。现在知道有这种情况就行。

▶ 边界 3：引用编号对应问题

  如果一个文档块在 Prompt 里是 [3]，但 LLM 引用了 [5]（这个编号根本不存在）？
  → LLM 在"幻觉引用"——它自己编了一个引用编号。

  处理方式：
    ① 代码层面检查：引用解析后，过滤掉不存在的编号
    ② Prompt 层面强化："只能引用参考资料中已有的编号 [1]-[N]"
    ③ 严重情况：如果幻觉引用太多，重新生成

八、今天的核心认知
===================

1. RAG 解决 LLM 幻觉：先检索真实资料 → 再让 LLM 基于资料回答。
   不是在 LLM 训练时塞资料，而是在推理时动态注入。

2. 引用标注让答案从"信不信由你"变成"查得到源头"。
   [1] [2] 是连接"答案"和"来源"的桥梁。

3. Prompt 模板是 RAG 生成质量的关键。一个好的模板包含：
   资料区（编号 + 原文）、约束区（只能基于资料）、引用指令、兜底规则。

4. 引用解析 = 正则提取 [N] → 映射原文档。逻辑很简单，但价值很大。

5. 边界情况要提前考虑：资料不足时怎么说？资料冲突时怎么处理？
   这些不是在代码出 bug 时才想，而是在设计 Prompt 模板时就覆盖。

6. 端到端 RAG 管道 = 检索 + 增强 + 生成 + 引用 → 一个函数搞定。
   这是你 Week 4 项目的核心引擎。
